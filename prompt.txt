You are a senior product engineer + architect. Design the first version (v0.1) of an all-in-one AI orchestration tool called “Langpedia”.

Goal
- Langpedia is a Python-first orchestration platform for AI apps: agents + RAG + MCP tool access + basic evaluation.
- It must have BOTH:
  1) a Next.js web UI (“Studio”)
  2) a terminal-first CLI (“langpedia”) that can run the same workflows locally or against a server.

Target users
- Developers who want a simple, unified way to build, run, debug, and deploy AI workflows.
- The UX must feel easy like a visual builder, but the underlying representation must be code-friendly and versionable.

Core product principles
- One source of truth: a Workflow Spec (JSON/YAML) that can be edited in UI, generated by CLI, and executed by backend.
- Deterministic execution tracing: every run produces an event stream (steps, tokens, tool calls, retrieved docs) viewable in UI and CLI.
- Safe-by-default: secrets management, tool permissions, sandboxed execution options.
- Minimal but complete: v0.1 focuses on “build → run → trace” for workflows with RAG and MCP tools; fine-tuning is NOT in v0.1.

Deliverables
Provide:
1) The MVP feature list (must-have, nice-to-have)
2) UX description + main screens for the Next.js Studio
3) CLI commands and UX examples (with output examples)
4) Backend architecture (Python) and API design (FastAPI)
5) The Workflow Spec format (YAML recommended) + examples
6) Execution engine design (graph/state machine) and event schema
7) Storage choices for v0.1 and what can be local-only (SQLite) vs server (Postgres)
8) Security model: auth, secrets, tool permissions, audit logs (minimal viable)
9) Repo structure (monorepo) with Next.js + Python + shared workflow spec/types
10) A realistic implementation plan in milestones (M0, M1, M2), each with acceptance criteria
11) Include 2 end-to-end example workflows:
    - “RAG QA over docs” (ingest → index → retrieve → answer with citations)
    - “Agent with MCP tools” (planner → tool call via MCP → final response)

Hard constraints
- UI: Next.js (App Router), TypeScript, Tailwind, React Flow (or similar) for graph editor.
- CLI: Python CLI using Typer (or Click). It must support local execution (no server required) and remote execution (server mode).
- Backend: Python FastAPI. Execution workers can be in-process for v0.1 (no Celery required).
- RAG: choose one vector store for v0.1 (Qdrant or pgvector). Provide justification.
- MCP: support “connect MCP server” + “list tools” + “call tool” from both UI and CLI.
- Observability: implement a basic built-in trace viewer (no external dependency required), but design it so Langfuse integration can be added later.
- No fine-tuning in v0.1.

What Langpedia should look like (UI)
- Projects page (workspaces)
- Workflow Studio (drag/drop nodes, config panel, run button)
- Runs page (list and filters)
- Run Trace page (timeline of steps + payload viewer)
- Data page (document upload + ingestion status + index health)
- MCP page (servers, tools list, permissions)

What Langpedia should look like (CLI)
- langpedia init
- langpedia project create <name>
- langpedia workflow new <name>
- langpedia workflow validate <file>
- langpedia run <file> --input <json> [--local | --remote <url>]
- langpedia runs list / langpedia runs show <run_id>
- langpedia rag ingest <path>
- langpedia mcp add <name> <url>
- langpedia mcp tools <name>
- langpedia mcp call <name> <tool> --args <json>

Mandatory output format
- Write your response as a structured spec with headings:
  1. Product scope v0.1
  2. Architecture
  3. Workflow spec (YAML)
  4. Event/trace schema
  5. UI screens
  6. CLI commands
  7. APIs (FastAPI endpoints)
  8. Repo layout
  9. Security and secrets
  10. Milestones + acceptance criteria
  11. Example workflows (full YAML + sample run output)

Tone and style
- Be precise and implementation-ready: include schemas, endpoint shapes, and examples.
- Use clear defaults and opinionated choices; avoid “it depends” unless necessary.
- Keep v0.1 lean and doable in a few weeks by a small team.
